# V2.2 PRD：稳定调色工作台

核心结论：
V2.2 不继续堆功能，优先解决“调整后看不见、调参没反应、AI 经常超时、布局不顺手、效果不可控”。
主要问题：
- 主预览容器不固定，图片和页面跳动。
- 图片队列在底部，选图上下文断裂。
- AI 未分析前展示伪建议。
- 并排/分割视图无法稳定同时显示原图和调整后。
- 参数区数值重复。
- 手动调参缺少生效反馈和错误态。
- AI 超时缺少继续等待、重试、错误分类。
- 缺少高光 clipping、阴影死黑、饱和断层检测。
- 预览和导出一致性未验收。
- 测试缺少 Canvas、AI 状态、参数作用范围覆盖。
V2.2 目标：
- 稳定看到原图与调整后。
- 调参 100ms 内触发预览更新并有状态反馈。
- 四种预览模式都稳定显示。
- AI 超时可恢复。
- 参数作用范围清楚。
- 图片队列上移，工作台顺序改为：素材选择 → 主预览 → 控制栏。
任务优先级：
- P0：预览显示、并排/分割、手动调参生效、AI 超时重试。
- P1：队列上移、参数区简化、AI 应用强度、高光风险检测。
- P2：预设中心、项目/历史记录、debug 日志面板。

### 调整方案细化（V2.2）

#### 1. 调整工作台的核心状态模型

- 每张图片维护独立的调整状态，包含：原图、当前参数、AI 建议参数、已应用参数、预览渲染状态、导出状态。
- 参数来源分为三类：手动调整、AI 建议、预设。三类来源可以覆盖同一组参数，但页面必须展示当前生效来源。
- 调整采用“即时预览、显式应用”的体验：用户拖动参数后 100ms 内更新预览；AI 建议需要用户点击应用后才进入当前参数。
- 当前参数是预览和导出的唯一数据源，避免预览一套、导出一套。
- 切换图片时保留各自参数，不把上一张图的调整误带到下一张图；如需批量应用，作为后续能力单独设计。

#### 2. 手动调参方案

- 参数区只保留一处数值展示：滑杆旁显示当前值，不再在卡片、摘要、标签中重复显示。
- 每个参数调整后进入“更新中”状态，预览成功后显示“已更新”，失败时显示“未生效，可重试”。
- 参数分组建议：光线、颜色、质感、细节。P0 先保证曝光、对比度、色温、色调、饱和度、高光、阴影能稳定生效。
- 每组提供“重置本组”，全局提供“重置全部”。重置后同样触发预览更新。
- 滑杆拖动过程中只更新本地参数和预览，不触发 AI 分析；AI 分析必须由用户主动发起。
- 参数边界需要明确，例如曝光、色温、饱和度等都要有最小值、最大值、步进和默认值，避免后续出现不同组件范围不一致。

#### 3. AI 调整方案

- AI 未分析前，不展示伪建议，只展示空状态和“开始分析”入口。
- AI 分析结果必须返回结构化参数、建议说明、风险提示和可应用强度。
- AI 建议默认不直接覆盖当前参数，用户可选择“应用 25% / 50% / 100%”或通过强度滑杆混合到当前参数。
- AI 应用后，页面需要标记“当前参数来自 AI 建议 + 用户手动微调”或“当前参数来自手动调整”，让来源可解释。
- AI 超时后保留当前预览和参数，提供“继续等待”“重新分析”“取消”三个动作。
- AI 错误分类至少包含：网络错误、服务超时、图片不可分析、返回格式异常、无可用建议。不同错误给不同恢复入口。

#### 4. 预览与对比方案

- 主预览容器固定尺寸和最小高度，图片加载、切换、参数更新时不能导致页面跳动。
- 四种预览模式统一使用同一套原图和当前参数：原图、调整后、并排对比、分割对比。
- 并排对比必须同时稳定显示原图和调整后，不能因为容器宽度不足只显示一侧。
- 分割对比需要固定可拖动分割线，拖动时只改变遮罩比例，不重新生成参数。
- 预览更新期间不清空画面，只在角落展示轻量状态，避免用户误以为图片丢失。
- 图片队列上移到素材选择区后，选中态、分析态、调整态都要在缩略图上可见。

#### 5. 高光、阴影、饱和风险检测

- 每次参数更新后运行轻量检测，输出三类风险：高光 clipping、阴影死黑、饱和断层。
- 风险提示不阻塞调整，只在预览旁显示风险徽标和对应参数建议。
- 高光风险优先提示降低曝光或高光；阴影风险提示提升阴影或降低对比；饱和风险提示降低饱和度或调整色温。
- P1 先做阈值检测和提示，不要求自动修复；自动修复可作为 P2 或后续版本。

#### 6. 参数生效与导出一致性

- Canvas 预览和导出共用同一套参数转换函数。
- 导出前需要重新读取当前参数，不允许使用旧缓存。
- 验收时使用同一张测试图分别对比预览截图和导出图，确认曝光、色温、饱和度等基础参数方向一致。
- 如果导出失败，不能丢失用户当前参数，需要允许重试导出。

#### 7. P0 落地清单

- 固定主预览容器，消除图片切换和渲染过程中的布局跳动。
- 修复原图、调整后、并排、分割四种预览模式的稳定显示。
- 建立当前参数单一数据源，保证手动调参 100ms 内触发预览更新。
- 增加调参状态反馈：更新中、已更新、失败可重试。
- 移除 AI 未分析前的伪建议展示。
- 增加 AI 超时后的继续等待、重新分析、取消。
- 增加基础错误分类，确保失败后用户能恢复当前工作。

#### 8. P1 落地清单

- 图片队列移动到素材选择区，并展示选中、分析中、已调整、失败状态。
- 简化参数区，移除重复数值和重复说明。
- 增加 AI 建议应用强度，支持 25%、50%、100% 或连续强度。
- 增加高光、阴影、饱和风险检测和提示。
- 增加参数分组重置和全局重置。

#### 9. 验收标准

- 任意手动参数变化后，100ms 内能看到预览进入更新状态，成功后画面发生可感知变化。
- 连续拖动滑杆 10 次，页面不跳动、不白屏、不丢失当前图片。
- 四种预览模式在同一张图片上切换 20 次，不出现原图和调整后错位、缺失或尺寸异常。
- AI 超时场景下，用户仍能继续手动调参，并能选择继续等待或重新分析。
- AI 未分析前页面不出现具体建议文案或虚假的参数结果。
- 调整后导出的图片与当前预览在基础色彩方向上保持一致。
- 切换图片后，每张图保留自己的调整参数和状态。

#### 10. 暂不做的内容

- 不做复杂批量套色和跨图片同步参数。
- 不做完整历史版本树，只保留当前参数、重置和必要的失败恢复。
- 不做自动修复高光、阴影、饱和问题，只做检测和提示。
- 不扩展新的 AI 风格库，优先把已有调整链路做稳定。

## Shotai V2.2 P0/P1 实施计划

### Summary

按 Notion PRD 先实现 P0，再实现 P1；不做 P2 的预设中心扩展、项目/历史记录、debug 日志面板。当前基线 npm test 已通过 8/8。核心策略是把“当前参数”作为预览和导出的唯一数据源，补齐 Canvas 渲染状态、AI 恢复路径、队列上移、AI 强度应用和风险检测。

### Key Changes

- P0 预览稳定性：重构 CanvasPreview.tsx，固定主预览尺寸，避免模式切换时 canvas ref 丢失；渲染中保留上一帧，增加 idle/loading-source/rendering/ready/error 状态反馈。
- P0 参数生效：在 App.tsx 增加参数来源模型 manual/ai/preset/mixed，手动调参 100ms 内进入预览更新状态；导出继续读取当前 currentAdjustments。
- P0 AI 恢复：把 AI 等待改成“软超时”交互，约 18s 显示继续等待/重新分析/取消；只有取消才 abort。错误分类为网络、服务超时、图片不可分析、返回格式异常、API 配置/额度、未知错误。
- P1 素材选择：将图片队列从底部移动到素材选择区，页面顺序变为素材选择 → 主预览 → 右侧控制栏；缩略图展示选中、已调整、分析中、导出失败状态。
- P1 参数区简化：在 AdjustmentPanel.tsx 保留滑杆 + 一个数值输入，移除重复 output；每个参数组增加“重置本组”。
- P1 AI 强度：AI 建议默认不覆盖参数，提供 25%、50%、100% 快捷应用和连续强度滑杆；应用公式为 blend(aiSnapshot, aiAdjustments, strength)。
- P1 风险检测：新增纯函数检测高光 clipping、阴影死黑、饱和断层；风险徽标放在主预览旁，只提示不自动修复。

### Interfaces And Types

- 新增 AdjustmentSource = 'manual' | 'ai' | 'preset' | 'mixed'，存到当前图片状态；手动改 AI 参数后标为 mixed。
- 扩展 BatchImageItem：增加 adjustmentSource? 和可选分析状态字段，用于缩略图状态展示。
- 新增 PreviewRenderStatus 与 PreviewRisk 类型；CanvasPreview 增加 onRenderStatusChange、onRisksChange 回调。
- 新增工具函数：blendAdjustments(base, target, strength)、resetAdjustmentGroup(values, groupId)、detectPreviewRisks(imageData, adjustments)。
- analysisApi 改为抛出带 code/status/message 的错误对象，UI 只根据 code 渲染恢复动作。

### Implementation Order

1. P0-1：重构 Canvas 预览生命周期和状态反馈，修复原图、调整后、并排、分割四种模式稳定显示。
1. P0-2：接入预览渲染状态到工作台 UI，确保拖动参数时显示“更新中/已更新/失败可重试”。
1. P0-3：移除 AI 未分析前的本地伪建议展示，保留空状态和“开始分析”入口。
1. P0-4：实现 AI 软超时、继续等待、重新分析、取消，以及错误分类。
1. P1-1：调整布局，把 BatchImageQueue 上移到素材选择区，补齐缩略图状态。
1. P1-2：简化参数面板，增加参数组重置。
1. P1-3：实现 AI 强度混合应用，替换当前“预览建议”直接写入参数的行为。
1. P1-4：实现预览风险检测和预览旁徽标。
1. P1-5：补齐测试、跑完整验证并做浏览器视觉 QA。

### Test Plan

- Unit：blendAdjustments 在 25/50/100 强度下结果正确；参数组重置只影响该组；风险检测能识别高光、阴影、饱和风险。
- Unit/API：AI 错误分类覆盖 timeout、network、bad response、quota/config、cancelled。
- Integration：切换四种预览模式 20 次不白屏；连续调参 10 次只保留最新渲染结果；切图后每张图保留独立参数来源。
- Visual QA：启动本地 dev server，用浏览器检查桌面和移动宽度下素材区、预览区、控制栏不重叠，canvas 非空，风险徽标不遮挡主图。
- Final verification：运行 npm run lint、npm test、npm run build；如 API server 相关改动存在，再运行 node --check server/shotai-api.mjs。

### Assumptions

- “继续等待”采用前端软超时，不引入后台 job 系统；真正失败后显示重新分析。
- AI 强度采用用户已选择的推荐方案：25/50/100 快捷按钮 + 连续滑杆。
- 风险提示采用用户已选择的推荐方案：放在主预览旁徽标。
- P1 风险检测只提示，不自动修复参数。
